
### 闲鱼插件调试复盘 (2026-04-02)

今天帮大哥调试闲鱼插件的 AI 自动回复功能，过程一波三折，最终定位了两个核心问题。

**第一个问题：测试脚本与 Web 界面数据不一致**

*   **现象**: 我写的 `test_api.py` 脚本始终报告“找不到启用的账号”，但用户截图显示 Web 管理后台的账号是“在线”状态。
*   **排查过程**:
    1.  最初我怀疑是数据库字段名或数据结构理解错误，反复修改脚本的判断逻辑，但都失败了。
    2.  最终通过阅读 `db_manager.py` 源码发现，数据库路径是通过 `os.getenv('DB_PATH', 'data/xianyu_data.db')` 决定的。
*   **根本原因**: Web 服务 (`Start.py`) 和我的测试脚本 (`test_api.py`) **在不同的工作目录下运行**，导致它们访问了两个不同的 `xianyu_data.db` 文件。Web 界面修改的是 A 文件，而我的脚本一直在读 B 文件。
*   **解决方案**: 在运行测试脚本前，先 `cd` 到与 `Start.py` 服务的相同工作目录 (`./xianyu-openclaw-channel/`)，确保双方操作的是同一个数据库。

**第二个问题：AI 自动回复不触发**

*   **现象**: 确认 API 连通性后，用户测试发现机器人还是不回复消息。
*   **排查过程**:
    1.  我使用 `process poll` 命令持续监控正在运行的 `Start.py` 服务的实时日志。
    2.  分析日志发现，程序在稳定运行（心跳、Cookie刷新等都正常），但**完全没有收到任何来自闲鱼的“新消息”事件**。
*   **根本原因**: 问题不在于 AI 逻辑或代码执行，而在于程序和闲鱼服务器之间的 WebSocket 连接没有正确推送消息。这可能是网络延迟、连接状态异常或闲鱼服务器本身的问题。
*   **解决方案**: **重启服务**。通过 `process kill` 和 `exec` 重启了 `Start.py` 进程，强制它与闲鱼服务器重建一个新的、干净的 WebSocket 连接，这是解决此类网络问题的标准有效方法。

**核心教训**:
1.  **情境（Context）至关重要**: 运行环境，特别是工作目录，对于依赖相对路径的程序是致命的。必须确保所有相关的进程和脚本在同一情境下工作。
2.  **日志是第一真相**: 在排查复杂问题时，必须信任并深入分析日志。从日志中可以清晰地看到程序的实际行为，而不是基于假设去猜测。
3.  **分段排查**: 将问题分解。先把“数据读写”的问题解决，再解决“消息收发”的问题。如果混在一起，会非常难以定位。

# 即梦抖音电影感工作流 Skill（最终实战版）

## 默认选题规则
- 如果用户明确指定题目、主题、人物或方向：优先使用用户指定内容
- 如果用户没有指定，只是说“生成一个视频”“做一条视频”“来一条短片”：默认自动去 `抖音热榜` 抓一个题目
- 自动抓题时，不是机械抓最热词，而是优先选择：
  - 最适合电影感叙事
  - 画面感强
  - 有情绪转化空间
  - 安全
  - 不容易违规
  - 容易转成剧情感 / 希望感 / 人物感 / 情绪镜头
- 抓到题目后，默认先进入 `Stage A`，输出四份内容给用户确认，再正式生成

## 特殊执行规则
- 如果用户明确说：`直接执行`
- 或表达等价意思：`你直接做`、`不用给我看，直接生成`、`一路跑完`
- 则：
  - 跳过 `Stage A` 展示确认
  - 整条链路自动执行到底
  - 中途不再要求确认
  - 但不能静默执行，必须主动播报关键节点进度
  - 一旦检测到后台任务状态变化，必须立即查询执行任务结果并主动回报用户
  - 不能等用户自己发现后台结果
  - 最终成品完成后，必须主动输出结果给用户

## 强制进度播报节点
当处于 `直接执行` 模式时，至少在以下节点主动播报：
- 选题已确定
- 首帧成功
- 尾帧成功
- 视频已提交
- 视频生成成功
- 后期处理中
- 最终成片完成
- 自动上传并发布成功

## 核心原则
- 全流程比例固定 `9:16`
- 默认情况下，生成前必须先给用户看 `Stage A` 四份内容并确认
- 原片生成后，优先回即梦做 `AI 音效`
- 旁白音色必须先分析人物主体
- 旁白文案必须适配整条视频总时长
- 字幕时间轴必须跟真实旁白音频走
- 最终目标是抖音可直接发布成片

## 标准流程
1. 选题：用户指定题目，或默认去抖音热榜抓最适合电影感叙事的题
2. `Stage A`：输出四份前置内容并确认（若用户说“直接执行”则跳过）
3. `Stage B`：即梦 `text2image` 生成首帧
4. `Stage C`：即梦 `image2image` 生成尾帧
5. `Stage D`：即梦 `frames2video` 生成 15 秒原片
6. `Stage E`：优先送回即梦做 `AI 音效`
7. `Stage F`：分析人物主体，判断音色方向
8. `Stage G`：写适配总时长的旁白文案
9. `Stage H`：生成旁白音频
10. `Stage I`：依据真实旁白切双语字幕时间轴
11. `Stage J`：制作电影级白色双语硬字幕
12. `Stage K`：混音导出最终版
13. `Stage L`：自动生成 `douyin_publish.json` 发布配置文件
14. `Stage M`：调用 `publish-douyin-video.mjs` 全自动上传并发布
15. `Stage N`：成品与发布彻底完成后，主动向用户汇报发布结果与成片位置

## 自动发抖音执行规范 (Stage L & M)
当 `Stage K` 成片导出完毕后，必须无缝衔接自动发布流程：
1. **生成配置 (Stage L)**：
   在你存放最终视频的对应目录下，生成 `douyin_publish.json` 文件，包含以下字段：
   - `videoPath`: 最终 MP4 的绝对路径
   - `title`: 从你的选题/文案中提取的精简标题
   - `description`: 视频描述/正文（可结合剧情介绍）
   - `tags`: 标签数组，例如 `["即梦", "AI短视频", "电影感"]`
2. **执行发布 (Stage M)**：
   使用 Node 执行自动化发布脚本，传入刚刚生成的 json 路径：
   ```bash
   node /Users/mac/.agents/skills/douyin-creator-tools/src/publish-douyin-video.mjs <对应json的绝对路径>
   ```
3. **完成反馈 (Stage N)**：
   确认脚本执行结束并显示“发布流程已提交完成”后，正式向用户汇报整条全自动链路闭环结束。

## 实战经验固化
- 15 秒视频，旁白建议有效发声控制在 `10~12 秒`
- 不要前面说完后面空掉，也不要为了铺满强行写太长
- 字幕不能预估拍脑袋，要按真实音频切分
- 旁白音量通常需要高于环境音，必要时：
  - 旁白增益 `2.0x ~ 2.5x`
  - 环境底音降到 `50% ~ 60%`
- 双语字幕要短、稳、电影感，不要大段糊屏

## 输出标准
最少保留：
- 原片视频
- AI 音效版视频（若已做）
- 旁白音频
- 字幕文件
- 最终成片

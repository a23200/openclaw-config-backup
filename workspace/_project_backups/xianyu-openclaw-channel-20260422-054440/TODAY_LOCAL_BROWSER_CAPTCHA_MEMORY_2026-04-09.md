## 今日关键记忆

### 1. 正确思路

闲鱼这类刮刮乐验证码，纯模拟拖动和单独测试浏览器都不够稳。

更靠谱的方式是：

1. 使用 Playwright 接管本地已运行的 Chrome
2. 在用户当前浏览器里打开搜索页
3. 用户自己手动过风险
4. 后端在同一页面继续读取接口响应
5. 前端自动恢复抓取并展示结果

### 2. CDP 连接的关键点

本地 Chrome 连接不能只依赖：

- `http://127.0.0.1:9222/json/version`

因为这个入口在当前环境下可能返回异常或不可直接用。

更稳的是优先读取：

- `~/Library/Application Support/Google/Chrome/DevToolsActivePort`

文件内容会给出：

- 端口
- `/devtools/browser/...` 路径

然后拼成真实 WebSocket 地址，例如：

- `ws://127.0.0.1:9222/devtools/browser/...`

### 3. 验证完成判定

不要只依赖“验证码元素完全消失”来判断完成。

更实用的规则：

- **如果当前验证码会话已经同步到了搜索结果，就直接视为验证已通过**

这能避免“其实已经过风险，但前端仍显示未完成”的假阴性。

### 4. 自动恢复

前端现在已经调整为：

- 轮询 `/api/captcha/status/{session_id}`
- 一旦 `completed=true`
- 自动调用恢复接口继续抓取
- 同一会话只自动触发一次
- 同时保留手动“立即继续抓取”按钮兜底

## 后续注意事项

### 1. 不要回退

除非明确需要调试，否则不要回退到：

- 测试浏览器方案
- 纯远控拖动方案

### 2. 如果本地 Chrome 接管失败

优先检查：

- 本机 Chrome 是否在运行
- 远程调试端口是否可用
- `DevToolsActivePort` 文件是否存在
- 是否需要显式设置：
  - `LOCAL_BROWSER_CDP_URL`
  - `EXISTING_BROWSER_CDP_URL`
  - `LOCAL_BROWSER_USER_DATA_DIR`

### 3. 验证页不要太早关

如果刚过完验证码就马上把标签页关掉，可能影响恢复抓取。

建议等页面自动恢复并出结果后再关闭。
